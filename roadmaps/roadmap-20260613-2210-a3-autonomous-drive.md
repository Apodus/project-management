## Phases (each: plan → adversarial verify → execute → commit)

### P1 — Drive-session skeleton (invoke /vision from the daemon)
A `DriveRunner` (mirror `ImplementRunner`'s injectable seam) that, for an `implement{systemic}` disposition, spawns a bounded headless session in an isolated worktree whose task is to PRODUCE A VISION for the escalation — invoking the `/vision` skill (or the equivalent inline orchestration: architect-synthesize → adversarial-verify → write `roadmaps/vision-<ts>-<slug>.md` + the PM epic/tasks). Define the drive-session input/result discriminated union (`vision_ready{visionPath, epicId}` | `give_up` | `error`), cross-checked against a real committed vision file + a real PM epic (never inferred from the sentinel, the A1 commit-cross-check discipline). NO campaign execution yet (P2). The loop routes `implement{systemic}` → `runDriveSession` (replacing the A1 `needs_human` placeholder) behind `auto_implement.enabled`. Tests: assess→systemic spawns a drive session (injected runner) that returns `vision_ready`; the vision file + epic existence is cross-checked; give_up/error → escalateToHuman; enabled=false ⇒ no drive session spawns; answer/bounded paths byte-identical.

### P2 — Run the campaign phases (reuse A1-implement + A2-land per phase)
Extend the drive session to DRIVE the produced vision as a campaign: invoke `/campaign` (or inline the commander pattern) so each phase runs plan→adversarial-verify→execute, and each phase's resulting branch lands via the SAME A2 verify-gated train + escalation-linked MR (the per-phase executor IS the A1 implement session; the per-phase land IS the A2 submit→train→post-back, escalationId threaded so each phase land is attributable to the escalation/arc). The drive result becomes `arc_complete{landedShas[], epicId}` | `arc_partial{landed, remaining, reason}` | `give_up` | `error`. Bounded by budget (A4 tightens; here just honor the existing spawn/concurrency budget). Tests: a small (scripted/injected) campaign whose phases each land via the train; phase lands carry escalationId; a mid-campaign phase reject → arc_partial (proven phases preserved, no discard); the commander discipline (fresh agents, checkpoint) is honored.

### P3 — Checkpoint / resume (daemon-restart survival)
The drive checkpoints its arc state (the campaign `*.progress.json` pattern + a durable pointer from the escalation/MR/epic to the in-flight vision path + completed phases) so a long arc survives a daemon restart. On restart, an in-flight drive is RESUMED from the next pending phase (not restarted) — reuse the A4 reclaim seam where it exists, else a drive-state file the loop reads on boot. The pendingLand/in-flight markers must reclaim-skip a live drive (no double-spawn). Tests: a mid-arc restart resumes from the next pending phase (not phase 1); a completed arc is not re-driven; the checkpoint round-trips; concurrent-drive guard (one drive per escalation).

### P4 — Escalation ↔ arc linkage + close-on-completion
The escalation tracks the arc end-to-end: the vision's PM epic + tasks link back to the escalation (a durable reference — reuse `postBackToEscalation`/addMessage metadata, or an epic↔escalation pointer); each phase land posts progress to the escalation thread; on ARC COMPLETION the escalation resolves (as the holder, the A2 resolve-as-holder path) with the shipped summary (the landed shas + the vision epic), and the origin auto-notices via C2. On arc give-up/budget-exhaustion → needs_human with the partial-progress payload (no proven work discarded). No-recursion: the arc's own landed MRs / any escalations they spawn never re-trigger auto-implement (the A2 structural seal extends — a drive-authored MR is escalation-linked + the no-recursion guards hold). Tests: a completed arc resolves the escalation + posts the summary + origin's undelivered cursor surfaces it; a give_up → needs_human + partial payload; no-recursion across the multi-phase arc.

Full-stack seal: assess→systemic→drive(vision→campaign phases→train lands)→escalation resolved→origin notified, against the injected runner + a scripted/real train land (extend responder-seal.test.ts or a server integration test). Confirm answer-mode + bounded-implement (A1) + A2 byte-identical. Docs (CLAUDE.md A3 paragraph / the responder README — the autonomous-drive capability, ships off). Full suite + build/typecheck/lint green; openapi/web regen if any REST surface changed. Commit; campaign-close checkpoint.
