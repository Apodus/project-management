# Campaign — Notes inbox: full-text view + smart promote (short topic + full content)

**Goal:** Fix two reported notes/inbox UX issues: (1) the web has NO way to read a note's full text (title clamped to 1 line, body to 2, no detail view/expander); (2) promote-to-proposal / promote-to-task prefill the whole `note.title` (often the entire captured text) into the new entity's title and leave the description empty — so the topic is bloated/cropped and content is effectively lost. The fix: a full-text note detail view, and a smart promote that picks a SHORT topic + carries the FULL content into the description so nothing is left out.
**Branch:** `campaign-notes-promote-fulltext` (off main). **Do NOT push** until the user approves the landing.

## The accurate current state (from the code map)
- **Note model** (`packages/shared/src/schemas/note.ts:69-96`, `server/src/db/schema.ts:1115-1153`): a note has `title: string (min 1, NOT NULL, one-line handle)` + `body: string | null (longer markdown detail)`. The model ALREADY supports short-title + full-body — the bug is upstream data shape (the whole text often lands in `title`) + the web surfacing.
- **No full-text view** (`packages/web/src/pages/notes-page.tsx`): `NoteCard` (451-592) clamps the title `line-clamp-1` (:500) and body `line-clamp-2` (:503-505); there is NO detail page/dialog/expander. The full text exists server-side (`GET /notes/{id}`, `useNote`/`api.getNote`) + via MCP `pm_get_note` (which renders the full body) — but the web never shows it.
- **Promote dialogs** (`notes-page.tsx` `PromoteToProposalDialog` 244-329, `PromoteToTaskDialog` 337-449): prefill `title = note.title` (:253/:346) and `description = ""` (:254/:347) — body is NOT carried in. On submit they send explicit `{title, description}`.
- **Promote API contract:** the REST routes (`server/routes/notes.ts` promote bodies 136-141/165-171) + the web client (`api.ts` 437-457) accept OPTIONAL caller-supplied `{title?, description?}`. The server (`note.service.ts` promoteToProposal 362-415 / promoteToTask 427-490) uses `finalTitle = title ?? note.title` and, when description omitted, auto-builds it from `note.body` + a provenance footer. So the server maps title→title, body→description correctly; the bug surfaces in the WEB prefill (whole title → title input, empty description).
- **Promote-to-task is human-only** (server 403 for non-humans `note.service.ts:438-444`; web gated `isHuman`). There is NO `pm_promote_note_to_task` MCP tool (web/REST-only). `pm_promote_note_to_proposal` exists (defaults title→note.title, description→body).

## Engineering values
No investment ceiling; reuse the existing dialog/card primitives + the note schema (no model change — title+body already exist); additive — a well-formed note (short title + body) promotes EXACTLY as today (the smart split only triggers when the title is pathologically long). The full-text view is purely additive surfacing. Match the existing notes-page idioms.

## The design (authoritative — resolve details in-phase)
**A shared derivation helper `deriveNotePromotion(note) → { title, description }`** (in `@pm/shared`, beside the note schema), the single source of truth for the smart split, used by BOTH the web prefill AND the server's title-omitted default:
- **title:** if `note.title` is a reasonable one-liner (≤ ~120 chars, no newline), use it VERBATIM (well-formed notes unchanged); else derive a SHORT topic — the first line, or the first sentence, else a word-boundary truncation to ~100 chars + "…".
- **description:** the full content so NOTHING is left out — if the title was shortened, prepend the FULL original `note.title`; then the `note.body` (if any). `[titleWasShortened ? note.title : null, note.body].filter(Boolean).join("\n\n")`. (Empty ⇒ undefined, preserving the existing "no description" behavior for an empty-body short-title note.)

## Phases (each: plan → adversarial verify → execute → commit)

### P1 — The shared `deriveNotePromotion` helper + smart promote (issue #1 end-to-end)
Add `deriveNotePromotion(note) → {title, description}` to `@pm/shared` (the only-if-long guard — well-formed notes byte-identical) + unit tests (short title verbatim + empty body ⇒ no description; long title ⇒ shortened topic + full title carried into description; long title + body ⇒ both in description; short title + body ⇒ title verbatim + body as description). Wire the WEB promote dialogs (`notes-page.tsx`) to PREFILL `title`/`description` from the helper (replacing `title=note.title, description=""`), so the user sees + can edit a short topic + the full content. ALSO apply the helper to the SERVER default (`note.service.ts` promoteToProposal/promoteToTask): when `title` is omitted, use the helper's short title; when `description` is omitted, build from the helper's description base (+ the existing provenance footer) — so the MCP/API path is fixed consistently (the only-if-long guard keeps well-formed promotes unchanged). Update the web dialog test (the `description: undefined` assertion at `notes-page.test.tsx:446` changes) + the server service tests + the MCP tool test if its default text changed. Tests: the helper unit table; the web prefill (short title + body-as-description); the server default (long-title note via API ⇒ short title + full content in description; well-formed note ⇒ unchanged).

### P2 — The full-text note detail view (issue #2)
Add a way to read a note's ENTIRE text in the web. RECOMMENDED: a **note detail dialog** (click the NoteCard — or a dedicated view/expand affordance — opens a Dialog showing the FULL title + FULL body [not clamped] + the metadata [kind, status, anchor/code-locator, severity, author, timestamps] + the promote/dismiss actions). Reuse the existing Dialog primitives + the `useNote`/`api.getNote` hook (or the already-loaded note). Keep the card's clamped preview; the detail view reveals everything. Decide: a dialog (recommended, idiomatic) vs an inline expander vs a route — pick the cleanest. Tests: clicking the card/affordance opens the detail showing the full body (not clamped); the metadata renders; the actions work from the detail. Seal: full suite + build/typecheck/lint green. Campaign close + docs note if warranted (likely none — UX fix).

## Done when
Both phases committed; an operator can READ a note's full text in the web (detail view) and PROMOTE it with a short topic + the full content preserved in the description (nothing left out), via the web buttons AND consistently via the MCP/API path; well-formed notes (short title + body) promote byte-identically; the whole suite + build/typecheck/lint green. No note-model change, no API surface change. Ships as a pure UX improvement.
