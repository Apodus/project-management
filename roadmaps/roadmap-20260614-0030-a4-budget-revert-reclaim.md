## Trust-first stance (every sub-agent)
We TRUST the agents + the autonomous drive+land. A4 adds NO human-approval gate. The merge-train verify gate remains the structural floor (main never breaks). A4's three levers are: (1) a **spend/concurrency budget** so a flood of escalations each spawning an arc is bounded for COST (not safety) — on exhaustion → pause + escalate (governance); (2) a **fast revert path** — trust-first means occasionally landing wrong, and a cheap revert through the same verify-gated train beats gating everything up front (insurance); (3) **reclaim** for stranded implement/drive sessions + submitted-but-unlanded MRs (recovery). Ships behind `auto_implement.enabled=false` (inert until the operator flips it).

## Phases (each: plan → adversarial verify → execute → commit)

### P1 — Spend + concurrency budget (extend the existing spawn budget)
A **budget** for auto-implement/drive that EXTENDS the responder's existing spawn-rate/concurrency seam: token budget + wall-clock + **max concurrent arcs** + **max arc duration** (the per-arc lifetime cap a multi-phase drive needs). Config under `auto_implement` (e.g. `budget: { max_concurrent_arcs, max_arc_duration_sec, token_budget?, ... }`) — canonical in the responder config (decide if any belongs in `@pm/shared`). On **exhaustion → pause new admission + escalate** (governance, NOT distrust): a new systemic/bounded disposition that would exceed the budget is held (the escalation goes needs_human or is re-queued with a budget-exhausted reason) rather than spawned. The max-arc-duration cap bounds a never-ending arc (the A3 risk: a runaway/never-completing drive) → on cap → escalate-to-human with the partial progress (no proven work discarded — the arc_partial path). Decide: does the budget live entirely responder-local (simplest, mirrors the existing spawn budget) or does any of it need a server-side surface? Default responder-local unless a concrete need. Tests: a disposition exceeding max-concurrent-arcs is held+escalated (not spawned); an arc exceeding max-arc-duration is capped→escalated with partial progress preserved; budget off/generous default ⇒ A1-A3 byte-identical; the existing spawn budget still governs bounded implements.

### P2 — Fast revert path (undo a landed_sha through the verify-gated train)
A **one-command revert**: a landed auto-fix/arc (a `landedSha`, or an arc's set of landed shas) judged wrong can be reverted by submitting a **revert branch through the same verify-gated train** (the train verifies the revert before main FFs — main never breaks even reverting). Surface it where an operator/human triggers it — decide the trigger (an MCP tool? a REST endpoint? a comment-driven action on the escalation/MR?). Reuse the A2 submit + the merge train; the revert MR is escalation/MR-linked for the audit chain (revert-of-<landedSha>). Trust-first: the revert is the INSURANCE that makes occasionally-landing-wrong cheap. NO new train machinery — a revert is an ordinary MR whose branch is `git revert <sha>`. Decide who produces the revert branch (a bounded responder session like the implement path, or a direct git-revert the loop/operator triggers). Tests: a revert of a landed_sha submits a verify-gated MR that lands the revert; the revert is linked to the original (audit chain); an arc-level revert (multiple shas) is handled or explicitly scoped to single-sha for v1.

### P3 — Reclaim for stranded implement/drive sessions + submitted-but-unlanded MRs
Extend reclaim (mirror 7.6.1's `reclaim-resolutions.ts` + the existing responder reclaim sweep) to recover: a **stranded implement/drive session** (a worktree slot leased but the session died — already partly handled by the slot `finally`; harden the cross-restart case) + a **submitted-but-unlanded MR** (a phase MR submitted but the train never landed/rejected it within a budget window → the arc stalls at re-park forever). On a stranded/stalled drive past `attempt_started_at + max_arc_duration + grace` → reconcile (if the work landed out-of-band → resolve) OR escalate (→ needs_human + comment). This closes the A3 "the drive getting stuck" risk the vision names. Tests: a stranded drive session past the cap → reclaimed (reconcile or escalate); a submitted-but-unlanded phase MR past the window → the arc escalates (not stuck forever); a normally-progressing arc is NOT falsely reclaimed.
