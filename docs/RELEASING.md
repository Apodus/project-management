# Releasing &amp; Publishing

Status of the versioning / npm-publishing arc, the release process, and the **parked** work needed to
actually publish. Written 2026-06-02 so this state isn't rediscovered later.

> **TL;DR** — Versioning is **live** (artifacts report their version). npm publishing is fully
> **scaffolded** but **parked**: it needs a one-time npm Trusted-Publisher registration + a version
> tag. No long-lived token is required (the release workflow uses OIDC).

### npm publish — scaffolded — commits `2c795e7` (scaffold) + `c28d62d` (scope → @urtela, OIDC)
- **Packages:** `@urtela/pm-mcp-server` and `@urtela/pm-integrator` (npm org **`@urtela`** — created by
  the user). The GitHub org is `Apodus`; the npm scope is `@urtela`. (An earlier draft used `@apodus`;
  fully renamed.)
- **Bundle-as-bin, zero runtime deps:** the esbuild bundle (`external: []`) inlines everything
  (`@pm/shared`, the MCP SDK, zod, commander, pino, simple-git), so all of those are now
  `devDependencies` and the *published* package has **no runtime dependencies**. `bin` points at the
  bundle; `files: ["dist/bundle/*.mjs", "README.md"]`; `prepublishOnly` rebuilds the bundle at pack time
  (`dist/` is gitignored).
- **Verified by dry-run** (`npm pack --dry-run`): each tarball is just `package.json` + `README.md` +

  Publishing (OIDC)**: `permissions: id-token: write`, `npm install -g npm@latest`, then

  automation tokens auto-expire and were friction).
- **Docs:** `docs/SETUP.md` §6 presents both distribution models — `npx @urtela/...` and the vendored

## ⛔ Parked — what's missing to actually publish

1. **Register a Trusted Publisher for each package on npmjs.com.** In each package's Settings →
   *Trusted Publisher* → GitHub Actions, set: organization `Apodus`, repository `project-management`,
   workflow `release.yml`. Do it for both `@urtela/pm-mcp-server` and `@urtela/pm-integrator`.
2. **Trigger a release** by pushing a `v*` tag **at a commit that has the publish scaffold** (≥ `2c795e7`).
   ⚠️ The existing `v0.1.0` tag points at `d8ecad7`, which is *before* the scaffold, so either:
   - re-point it: `git tag -f v0.1.0 <commit ≥ 2c795e7> && git push -f origin v0.1.0`, **or**
   - cut a fresh `v0.1.1` (bump the versions first — see "Process" below).

### Caveats to settle at publish time (don't pre-solve)
- **First-publish bootstrapping:** npm Trusted Publishing can pre-register a not-yet-existent package,
  but depending on your npm account/CLI the *very first* publish may still need a one-time **granular**
  token. Check npm's current docs when you go to publish.
- **npm version:** OIDC trusted publishing needs npm ≥ 11.5.1 (the workflow upgrades via
  `npm install -g npm@latest`).
- **`pnpm` vs `npm` for publish:** this is a pnpm *workspace*, but the release workflow uses
  `npm publish` per-directory (most reliable OIDC/provenance support). Don't switch it to `npm publish -w`
  (that needs npm-workspaces config, which this repo doesn't use).
- **LICENSE not in the tarball (minor):** only a root `LICENSE` (MIT) exists; the published packages
  don't include it. To ship it: add a per-package `LICENSE` + `"LICENSE"` to each `files` array.
- **The integrator over `npx`:** the integrator is a long-lived daemon; vendoring the bundle (via
  `scripts/distribute.mjs`) is usually preferable to `npx` for it. The MCP server is the natural
  `npx` candidate.

## Process (how releasing works once unparked)

1. Make changes; update `CHANGELOG.md` under a new `## [x.y.z]` heading.
2. Bump the version (lockstep) in the root + all `packages/*/package.json` (and keep the CHANGELOG/tag in
   sync). The artifacts pick the version up automatically via the generated `version.ts`.
3. Commit, then tag: `git tag -a vX.Y.Z -m "vX.Y.Z" && git push origin main vX.Y.Z`.
4. The tag push runs `release.yml` → builds the bundles → `npm publish --provenance` both packages via
   OIDC. No token.
5. (If clients vendor instead of `npx`) re-run `scripts/distribute.mjs` to ship the new bundles.
